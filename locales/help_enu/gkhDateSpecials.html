Dates special features

To modify date of an event you use "Edit events and materials" dialog window.

GEDKeeper supports the following date subtypes: "Exact", "Before", "After", "Between", "Period before", "Period after", "Period between", "Approximately", "Calculated", "Evaluated".

  • Subtypes "Before", "After", "Between", "Period before", "Period after", "Period between" specify range of dates,
  • Subtypes "Approximately", "Calculated", "Evaluated" specify inaccurate or relative dates.

Dates can be incomplete, i.e. one or more parts of date may be skipped. To the extent that you can define only day number, or month only, or year only. Of course, you can combine such parts.

For a date you entered you can specify type of calendar, where the date is calculated. GEDKeeper supports Gregorian, Julian, Jewish, French and Roman calendars. You can also specify that date is a date of Anno Domini period (BC is off) and a date of Before Christ period (BC is on). A note for the current version of GEDKeeper: in order to convert dates of two different calendars you have to use "Calendar" plug-in.

There are three operations in the program to handle dates: storing, showing and processing. Each operation has its own specific and that is why all these operations were separated from each other.

  • Storing. All dates are stored using special GEDCOM format. This allows user to define incomplete dates in any form,
  • Showing. When GEDKeeper needs to show you a date it translates date in internal form to a string view, that may include special symbols (if they were enabled in the program settings). For example, "Approximately 1701, January" date will be rendered as "__.01.1701" (when special symbols are off) or as "~ __.01.1701" (when special symbols are on) in columns of general lists. Dates range like "Between 01.01.1700 and 20.01.1700" will be shown as "01.01.1700 - 20.01.1700". "Period before 10.10.1800" date will be rendered as "< 10.10.1800" (special symbols are on).
  • Processing. When GEDKeeper processes dates (for example, makes arithmetical operations on dates or compares date) it translates them into special internal form ("Julian day"), that allows code compares and handles incomplete dates of different calendars.

When GEDKeeper shows an incomplete date it replaces unknown parts with "_" symbol. Some users opted against it. But such symbols allow to show clearly what parts of a date are missed. This also guarantees unified presentation of all dates through the program.

Incomplete dates are also allowed in filters on persons for birthdays and obits. Therefore, it is possible to select all men bore between 1 700 and 1 710, without specifying dates and months.

To get specific character of processing dates and events, it is necessary to introduce both the topics, as they are handled in the program. Date of an event is just a combination of known day, month and year within the bounds of some calendar. Date may be empty or incomplete, exact, relative or be a range. Date does not bear some other (non-calendar) information. Event is a record about certain case happened in man's life. Such a record includes type of event, date, place, description and additional information. Person entry, for example, can have a record about death with empty date. This means that man died but no one knows when. Therefore, GEDKeeper processes differently persons with record about death and persons without such record.

Some other software does not handle death and birth events by list like GEDKeeper does. They usually use "Died" flag and fields that store date and place of death. From the point of view of GEDKeeper all events are the same - GEDKeeper processes them the same way and never limits number of stored events. For purposes of the next researches, a single person record is able to store several death and birth events simultaneously (thus allowing to store several versions). This makes software more flexible: user does not have to write such events into the text notes or write birth and death events as range of dates. Moreover, having such approach user can attach source of information to each event and, later, see all the sources. When researcher gets a reliable information he can delete incorrect event or mark it as erroneous. The latter allows always to see that some information, even it was erroneous, was rechecked and refuted.

Number of years in "Age" and "Life span" columns get calculated as follow:

  • Life span is calculated strictly as difference between birthday and obit. Therefore, when one of these days is missed, nothing is done. Result is undefined.
  • Age field calculated as differences of dates between birth and death events (this is an age when a man died if death event has valid date). Age is not calculated when death event exists but has no date. Age is calculated using the current date when death event was not recorded (just event itself, not its date). For example, for a person was born in 1 700 and having no death event, in 2 015 GEDKeeper shows its "Age" as 315 years. To avoid this for men have died indeed (even if you do not know when this happened), always add death event with empty date.

To make processing of data bulk more easily (for example, for data taken from audit tale, having many records without specified obit) GEDKeeper offers check database service (accessible from "Service\Tools" menu, "Check database" tab). This service checks all person records and when it finds a record having no death event, with age exceeding the most well-known and scientifically approved limits of a human life, the service suggests an automatically fix for death event (user must select a suggestion and click "Fix" button). For GEDKeeper limit of a human life is 122 years (see "List of the verified oldest people" Wikipedia page). It should be noted that the checking function does not set obit of a person to the maximum value. It just creates an empty death event, just an indication.
